## What is typegen?

Typegen is a small CLI that introspects your running CopilotKit runtime and emits a TypeScript declaration file. Once generated, every `agentId` prop in `@copilotkit/react-core/v2` is constrained to the agents your runtime actually exposes — autocomplete works, typos fail type-check.

```bash
npx copilotkit typegen http://localhost:3000/api/copilotkit
```

Without typegen, `agentId` is just `string`. With typegen, it becomes a union of the agent IDs your runtime registered.

<Callout type="info">
  Typegen currently only constrains **agent IDs**. Tool names registered via
  `useFrontendTool`, `useRenderTool`, and friends remain unconstrained — you
  are free to register any client-side tool name, regardless of what the
  runtime exposes.
</Callout>

## How it works

```text
Your Runtime (running server)
  |  GET /info
  v
copilotkit typegen <url>
  |  reads agent names
  v
.copilotkit/types.ts       — type Agents = { ... }
.copilotkit/register.d.ts  — augments Register interface
  |
  v
TypeScript picks up the augmentation; AgentId resolves to your agent union.
```

### Step 1 — Runtime exposes agent info

Every CopilotKit runtime ships with a `/info` endpoint that returns the agents registered on the server, including their name and description.

### Step 2 — CLI generates types

Running `copilotkit typegen` writes two files into `.copilotkit/` at your project root:

#### `.copilotkit/types.ts`

```ts
// Auto-generated by @copilotkit/cli typegen
// Do not edit manually

export type Agents = {
  "default": {
    description: "Default conversational agent";
  };
  "planner": {
    description: "Plans and schedules events";
  };
};
```

#### `.copilotkit/register.d.ts`

```ts
import type { Agents } from "./types.js";

declare module "@copilotkit/react-core/v2" {
  interface Register {
    agents: Agents;
  }
}
```

The CLI also tries to update two project files for you:

- **`.gitignore`** — `.copilotkit/` is appended (generated files should not be committed).
- **`tsconfig.json`** — `.copilotkit/register.d.ts` is added to the `include` array, with comments and trailing commas preserved via [`jsonc-parser`](https://www.npmjs.com/package/jsonc-parser).

<Callout type="warn" title="When the CLI does NOT touch your tsconfig">
  The tsconfig update is skipped (and a message is printed) when the CLI can't be confident it would leave a working build behind:

  - **`tsconfig.json` is missing.** Add `"include": [".copilotkit/register.d.ts"]` manually.
  - **Parse errors.** `jsonc-parser` reported one or more errors. The CLI refuses to modify a config with syntax errors — fix the file first, then re-run.
  - **`extends` + no own `include`.** Adding an `include` here would override the base config's include entirely (this also covers explicit `include: []`, the canonical "clear the base" pattern). Add `.copilotkit/register.d.ts` to the `include` of the tsconfig that extends your base, or to the base itself.
  - **`include` is not an array.** If your config has e.g. `"include": "src/**/*"` (a bare string), the CLI refuses to overwrite it. Fix `include` to be an array of strings, then re-run.

  When you see one of these messages, the generated `.copilotkit/types.ts` and `.copilotkit/register.d.ts` are still written — only the tsconfig wiring is skipped. Type-safe agent IDs become active as soon as TypeScript picks up `register.d.ts` via the `include` (or any equivalent mechanism your project uses).
</Callout>

### Step 3 — Hooks pick up the augmentation

```tsx
import { useAgent, useFrontendTool } from "@copilotkit/react-core/v2";

// agentId is now "default" | "planner" — autocomplete + compile-time check
const { agent } = useAgent({ agentId: "planner" });

// Tool name stays unconstrained — register any client-side tool you like
useFrontendTool({
  name: "anything",
  description: "Client-side only — runtime doesn't need to know",
  agentId: "default", // <- still type-checked
  handler: async () => {},
});
```

## Multiple runtimes

Pass a comma-separated list of URLs to merge agent IDs from multiple runtimes:

```bash
npx copilotkit typegen http://localhost:3000/api/copilotkit,http://localhost:4000/api/copilotkit
```

If the same agent ID is exposed by multiple runtimes, the last one wins (with a warning logged).

## When to re-run

Re-run `copilotkit typegen` whenever you add, remove, or rename an agent on the server.

## CI enforcement

The generated `.copilotkit/` directory is auto-added to `.gitignore`, so the generated files don't ship in source control. To verify in CI that your frontend stays in sync with the deployed runtime, run the CLI against the deployed URL and gate on its exit code:

```bash
npx copilotkit typegen https://your-runtime.example.com/api/copilotkit
```

The CLI exits non-zero when:

- No agents are discovered at all.
- Any of the supplied runtime URLs fail to respond (even if other URLs succeeded — partial sync would silently drop agents from the failing runtime).
- The project root cannot be located, or generated files cannot be written.

Wire that exit code into your CI job to block deploys whose runtime exposes agents the frontend was never typed against.
