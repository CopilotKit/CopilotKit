## What is typegen?

Typegen is a CLI tool that introspects your running CopilotKit runtime and generates TypeScript types for your frontend. Once generated, every hook in your app becomes type-safe: agent IDs autocomplete, tool names are constrained per-agent, and handler/render parameter types are inferred automatically.

```bash
npx copilotkit typegen http://localhost:3000/api/copilotkit
```

Without typegen, hooks like `useFrontendTool` accept any string for tool names and type parameters as `Record<string, unknown>`. With typegen, the same hooks constrain names to your actual tools and infer parameter types from their schemas.

## How it works

### The pipeline

```
Your Runtime (running server)
  |  GET /info
  v
CLI: copilotkit typegen <url>
  |  reads agents, tools, parameter schemas
  v
.copilotkit/types.ts       <-- per-agent tool maps + merged Tools union
.copilotkit/register.d.ts  <-- augments Register interface in @copilotkitnext/react
  |
  v
TypeScript picks up the augmentation
  |
  v
Hooks infer types from Register --> type-safe agent IDs, tool names, parameters
```

### Step 1: Runtime exposes agent info

When your runtime starts, each agent registered in `CopilotRuntime` is queryable via the `/info` endpoint. For each agent, the runtime collects:

- **name** and **description** from the agent instance
- **tools** with their name, description, and JSON Schema parameters (converted from Zod schemas)

For example, a `BuiltInAgent` with tools defined via `defineTool()` automatically exposes tool schemas through its `getToolDefinitions()` method.

### Step 2: CLI generates types

Running `copilotkit typegen` produces two files in a `.copilotkit/` directory at your project root.

#### `.copilotkit/types.ts`

Contains per-agent tool maps and a merged `Tools` type. When multiple agents define a tool with the same name but different parameters, the merged type becomes a union:

```ts
// Auto-generated by @copilotkitnext/cli typegen
// Do not edit manually

export type Agents = {
  "default": {
    tools: {
      "getWeather": { city: string; unit?: string };
      "searchDocuments": { query: string; maxResults?: number };
    };
  };
  "planner": {
    tools: {
      "createEvent": { title: string; date: string; attendees?: string[] };
      "getWeather": { city: string; days?: number };
    };
  };
};

// "getWeather" exists on both agents with different params -> union
export type Tools = {
  "getWeather": { city: string; unit?: string } | { city: string; days?: number };
  "searchDocuments": { query: string; maxResults?: number };
  "createEvent": { title: string; date: string; attendees?: string[] };
};
```

#### `.copilotkit/register.d.ts`

Augments the `Register` interface in `@copilotkitnext/react` with the generated types:

```ts
import type { Agents, Tools } from "./types";

declare module "@copilotkitnext/react" {
  interface Register {
    agents: Agents;
    tools: Tools;
  }
}
```

The CLI also automatically:
- Adds `.copilotkit/` to your `.gitignore` (generated files should not be committed)
- Adds `.copilotkit/register.d.ts` to your `tsconfig.json` `include` array

### Step 3: Hooks become type-safe

Once Register is augmented, conditional types in the React package activate. Every hook benefits:

| Hook | What gets typed |
|------|----------------|
| `useFrontendTool` | `name` autocompletes tool names; `handler(args)` gets the correct parameter type |
| `useHumanInTheLoop` | `name` autocompletes; render props are typed |
| `useComponent` | `name` autocompletes; component props match the tool's parameters |
| `useRenderTool` | `name` autocompletes (including `"*"` wildcard); render `parameters` are typed |
| `useAgent` | `agentId` autocompletes to registered agent IDs |

## Type resolution

The type system resolves parameters using a priority chain:

1. **Explicit Zod schema** -- If you pass `parameters: z.object({...})` to a hook, `z.infer` from the schema always wins
2. **Register inference** -- If no schema is given and typegen has run, types are resolved from the generated Register
3. **Fallback** -- If Register is not augmented, names fall back to `string` and parameters to `Record<string, unknown>`

### Agent scoping

When `agentId` is provided, tool names and parameters are scoped to that specific agent:

```tsx
useFrontendTool({
  name: "createEvent",  // autocomplete: "createEvent" | "getWeather"
  agentId: "planner",
  handler: async (args) => {
    // args: { title: string; date: string; attendees?: string[] }
  },
});
```

Without `agentId`, all tools across all agents are available. Shared tool names produce parameter unions:

```tsx
useFrontendTool({
  name: "getWeather",
  handler: async (args) => {
    // args: { city: string; unit?: string } | { city: string; days?: number }
  },
});
```

### Frontend-only tools

Tools defined only in the frontend (not on any backend agent) are fully supported. The name constraint accepts any string while still providing autocomplete for registered names. Provide an inline `parameters` schema to get full typing:

```tsx
useFrontendTool({
  name: "showNotification",  // not in Register -- works fine
  parameters: z.object({ message: z.string() }),
  handler: async ({ message }) => {
    // message: string -- inferred from the schema
    alert(message);
  },
});
```

### Wildcard renderers

`useRenderTool` supports `name: "*"` as a fallback renderer for any tool call. When Register is augmented, the wildcard's `parameters` type resolves to the union of all possible tool parameters (scoped by agent if `agentId` is provided):

```tsx
useRenderTool({
  name: "*",
  agentId: "planner",
  render: ({ name, status, parameters }) => {
    // parameters is the union of all planner tool params:
    //   { title: string; date: string; attendees?: string[] }
    // | { city: string; days?: number }
    return <div>{name}: {status}</div>;
  },
});
```

## Quick start

<Steps>

### Start your runtime

Make sure your CopilotKit runtime is running and accessible:

```bash
npm run dev
```

### Run typegen

Point the CLI at your runtime URL:

```bash
npx copilotkit typegen http://localhost:3000/api/copilotkit
```

The CLI will print discovered agents and tools, then generate the type files.

### Verify tsconfig

The CLI automatically adds `.copilotkit/register.d.ts` to your `tsconfig.json`. Verify it's there:

```json title="tsconfig.json"
{
  "include": [
    // ... your existing includes
    ".copilotkit/register.d.ts"
  ]
}
```

### Use type-safe hooks

Your hooks now have full type inference. Restart your IDE or TypeScript server to pick up the new types.

```tsx
useFrontendTool({
  name: "getWeather",  // autocompletes from your actual tools
  agentId: "default",  // autocompletes from your actual agents
  handler: async (args) => {
    // args.city is typed as string
    console.log(args.city);
  },
});
```

</Steps>

## Multiple runtimes

If your app connects to multiple runtimes, pass comma-separated URLs:

```bash
npx copilotkit typegen http://localhost:3000/api/copilotkit,http://localhost:4000/api/copilotkit
```

Agents from all runtimes are merged into a single set of generated types. If two runtimes expose an agent with the same ID, the last one wins.

## Re-running typegen

Re-run the command whenever you:

- Add, remove, or rename agents
- Add, remove, or modify tool definitions (names, parameters)
- Change which tools are assigned to which agents

The generated files in `.copilotkit/` are gitignored by default. Run typegen as part of your dev setup or CI pipeline.
