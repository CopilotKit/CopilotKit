---
title: Posting arbitrary JSX as images
description: Post any app component — a shadcn card, a dashboard tile, a chart — to a thread with thread.post(<X/>). Arbitrary JSX renders to an image via Takumi (no browser); channels-ui JSX keeps rendering natively.
icon: "lucide/Component"
---

**Where you're starting from:** `thread.post` only understands the `@copilotkit/channels` JSX vocabulary (`<Message>`, `<Header>`, `<Section>`, …). A styled dashboard card or any other component from your own app has no way onto the thread short of screenshotting it yourself.

**Where you're headed:** `thread.post(<MrrCard/>)` just works — the SDK renders your app's own React component to a PNG (fonts and compiled CSS included) and uploads it, no wrapper or extra API required. Native channel components keep rendering exactly as before.

<Steps>
  <Step>
    ### How it's routed

    `thread.post` classifies whatever you pass it by **where it's authored**, not by inspecting it at call time:

    - A **plain React element built in an ordinary app file** (react-pragma, e.g. `<div>`, a shadcn card, `<MrrCard/>`) → routed straight to the **image** path via [Takumi](https://github.com/takumi-rs/takumi) and uploaded through the existing `postFile` path. The component is not called to "peek" at it — a plain element goes to the image renderer as-is.
    - A `@copilotkit/channels` component (`<Message>`, `<Button>`, …), or your own component **referenced under the channels JSX pragma** (or branded as a channels-ui component) that returns channel nodes (e.g. an `IssueCard` that returns `<Message>…</Message>`) → the engine calls the component, inspects what it returns, and renders **natively**, unchanged.

    ```tsx
    // Native — authored under the channels pragma (or branded), still renders
    // as real Block Kit / Adaptive Cards
    await thread.post(
      <Message accent="#16a34a">
        <Header>Deploy complete</Header>
        <Section>Shipped <b>v2.4.0</b> to production.</Section>
      </Message>,
    );

    // Arbitrary app JSX — a plain React element from an app file, routed to an
    // image automatically, no wrapper needed
    await thread.post(<MrrCard value="$48,200" delta={12} />);
    ```

    The "call the component and inspect its output" check only happens for components referenced under the channels pragma; a plain element authored in an app (react-pragma) file never gets called — it goes straight to the image path. A channels-pragma/branded component that throws (e.g. one that calls a hook) is treated as native rather than crashing the classifier. Classification may invoke such a component an extra time (peek, then bind on the native path), so components referenced under the channels pragma must be pure / side-effect-free.

    <Callout type="info" title="One post, one route">
      A single `post()` call is either native IR or one image — you can't mix native components and image output in the same message. Call `post()` twice if you need both.
    </Callout>
  </Step>
  <Step>
    ### Configure fonts and styling once

    Takumi is a static, browser-free rasterizer with **no system fonts** — only a built-in Latin (Geist) fallback. Register your app's fonts and compiled CSS once on `createChannel({ render })`, and every image post picks them up:

    ```ts title="bot.ts"
    import { createChannel } from "@copilotkit/channels";
    import { readFileSync } from "node:fs";

    const bot = createChannel({
      adapters: [/* ... */],
      agent: (threadId) => /* ... */,
      render: {
        fonts: [
          { name: "Inter", data: readFileSync("./fonts/Inter-Regular.woff2"), weight: 400 },
          { name: "Inter", data: readFileSync("./fonts/Inter-SemiBold.woff2"), weight: 600 },
        ],
        stylesheets: [readFileSync("./styles/global.css", "utf8")], // compiled Tailwind
        width: 720, // default canvas width; height is 480 unless you set one
      },
    });
    ```

    `stylesheets` accepts your compiled CSS as plain strings — class selectors and `@media` resolve. This example loads it with `node:fs` (consistent with the fonts above) so it runs as-is under Node/tsx. If your build already emits the compiled CSS as an importable string (e.g. Vite's `?inline` suffix), that's a **bundler-only** variant — it won't run under plain `tsx`/`node`:

    ```ts title="bot.ts (bundler-only variant, e.g. Vite)"
    import appCss from "./styles/global.css?inline"; // compiled Tailwind, resolved by the bundler

    // ...same createChannel call as above, with stylesheets: [appCss]
    ```

    <Callout type="warn" title="No fonts configured means Latin-only">
      Takumi can't detect missing glyphs ahead of render, so the SDK warns once when no `fonts` are configured — non-Latin text will drop from the image. Any other Takumi render error is surfaced (not swallowed), so a broken font or stylesheet fails loudly instead of silently.
    </Callout>
  </Step>
  <Step>
    ### Override per post

    Every `render` field can be overridden for a single post — useful for a wider dashboard, a taller card, or an extra font just for that message:

    ```tsx
    await thread.post(<WideDashboard data={rows} />, { width: 1200 });
    await thread.post(<CjkReport data={rows} />, { fonts: [notoSansJP] });
    await thread.post(<MrrCard value="$48,200" delta={12} />, {
      filename: "mrr.png",
      title: "MRR",
      altText: "Monthly recurring revenue card",
    });
    ```

    `filename`, `title`, and `altText` are upload metadata passed straight through to `postFile` — they don't affect rendering.
  </Step>
  <Step>
    ### Built-in chart components

    `@copilotkit/channels/charts` ships a set of presentational chart components that render through the same image path — no Recharts, no Chart.js, no DOM:

    - **Box-model:** `BarChart`, `StackedBar`, `Meter`
    - **SVG-based:** `LineChart`, `PieChart`, `Scatter`, `Sparkline`

    ```tsx
    import { BarChart, LineChart, PieChart, Sparkline } from "@copilotkit/channels/charts";

    // Uses the fixed default palette: #6366f1, #22c55e, #f59e0b, #ef4444, #06b6d4
    await thread.post(
      <BarChart
        title="Signups / day"
        data={[
          { label: "Mon", value: 120 },
          { label: "Tue", value: 240 },
          { label: "Wed", value: 180 },
        ]}
      />,
    );

    // Styled with an explicit line color + a dark card background. `LineChart`
    // is single-color (see note below), so `colors` takes exactly one entry.
    await thread.post(
      <LineChart
        title="MRR trend"
        data={mrrSeries}
        colors={["#2563eb"]}
        gridColor="#334155"
        style={{ backgroundColor: "#0f172a" }}
        labelClassName="text-muted-foreground text-xs"
        showGrid
      />,
    );

    // Multi-color example: BarChart/StackedBar/PieChart cycle the palette per
    // category, so a multi-entry `colors` array actually produces multiple colors.
    await thread.post(
      <PieChart
        title="Traffic by source"
        data={[
          { label: "Organic", value: 55 },
          { label: "Referral", value: 25 },
          { label: "Direct", value: 20 },
        ]}
        colors={["#2563eb", "#16a34a", "#f59e0b"]}
      />,
    );

    // Inline sparkline inside your own arbitrary card
    await thread.post(
      <div className="p-4 flex items-center gap-3">
        <span>Latency</span>
        <Sparkline data={p95} width={120} height={28} colors={["#dc2626"]} />
      </div>,
    );
    ```

    Every chart component shares the same `ChartStyleProps`: `className`, `style`, `width`, `height`, `colors`, `gridColor`, `labelClassName`, `title`, and `showGrid` — but not every field is honored by every chart:

    - **Colors:** Takumi does not resolve CSS `var()`/theme tokens for SVG `stroke`/`fill` (or for `backgroundColor` on box-model charts), so the default palette is a **fixed** set of hex values, not your shadcn `--chart-1..5` tokens. Pass your own `colors` prop to theme a chart explicitly.
    - **Single-color vs. palette-cycling:** `LineChart`, `Sparkline`, `Scatter`, and `Meter` are **single-color** — they only ever use `colors[0]`, so a multi-entry `colors` array has no extra effect on them. `BarChart`, `StackedBar`, and `PieChart` **cycle the palette per category** — pass a multi-entry `colors` array to get a distinct color per bar/slice.
    - **Background:** every chart hardcodes an inline `backgroundColor` (white, on the SVG canvas or the box-model root). Inline `style` always wins over a CSS class, so theming the background via `className` (e.g. a `bg-card` utility class) has **no effect** — the only way to override a chart's background is the `style` prop, e.g. `style={{ backgroundColor: "#0f172a" }}`.
    - **`width`/`height`:** honored by the SVG charts (`LineChart`, `PieChart`, `Scatter`, `Sparkline`) as the canvas size. `BarChart`/`StackedBar` fill their container and ignore both. `Meter` also fills its container width-wise and ignores `width`, but honors `height` as the bar's thickness.
    - **`gridColor`/`showGrid`:** only apply to `LineChart` and `Scatter`.
    - **`title`:** honored by `BarChart`/`StackedBar`/`LineChart`/`PieChart`/`Scatter`, not by `Meter`/`Sparkline`.
    - **Non-negative values:** `BarChart`/`StackedBar` assume non-negative values — a negative value clamps to a zero-height bar rather than rendering below the axis.
    - **Y-axis baseline:** `LineChart` anchors its y-axis at zero (its scale spans from `min(0, ...values)` to `max(...values)`), so a chart with only positive values always shows the zero baseline. `Sparkline` has no such anchor — it scales purely to the data's own min/max, so its shape communicates relative movement, not absolute value.

    These are separate from the existing **native** `<Chart>` component, which still renders as a real Block Kit/Adaptive Card chart where the surface supports it. The `charts` components are for image output styled with your app's own CSS.
  </Step>
  <Step>
    ### Know the limits

    Takumi is a **static SSR rasterizer** — it calls your components with server-render semantics, not a real browser. Keep this in mind:

    - **No runtime-JS chart or diagram libraries.** Recharts, Chart.js, and Mermaid all measure or paint the live DOM (`useEffect`, canvas, layout measurement) — none of that runs under Takumi. Use the built-in `charts` components (or your own pure-presentational markup) instead.
    - **`update()` and `postEphemeral()` don't accept arbitrary JSX.** An uploaded image can't be edited in place, so both throw a clear error if you pass them a React element or an unbranded component that returns one — post a new image instead of trying to update an existing one.
    - **Image posts are file-backed.** They return a `MessageRef` like any other post, but with more limited edit support than a native message.
    - **The canvas has a fixed size.** It defaults to 720×480; pass `width`/`height` (globally via `render`, or per post) for anything else.
  </Step>
</Steps>

**You've reached point B.** Arbitrary app components render to images alongside your native channel UI, styled with your own fonts and compiled CSS, with no browser anywhere in the process.
