## Execution tracing

The Mastra bridge forwards Mastra's own tracing through, so runs show up in your
Mastra observability backend with no extra wiring.

- **Inbound** — pass `tracingOptions` on your agent config to anchor each run
  under a caller-chosen trace. The shape is structural (no `@mastra/core`
  import): `{ traceId?: string; metadata?: Record<string, unknown> }`. It is
  threaded into `agent.stream()` / `resumeStream()` on the local, remote, and
  resume paths.

  ```ts
  getLocalAgents({
    // ...your agents
    tracingOptions: {
      traceId: myTraceId,
      metadata: { feature: "support-chat", tenant: tenantId },
    },
  });
  ```

- **Outbound** — the execution `traceId` Mastra assigns to a run is surfaced on
  the `RUN_FINISHED` event's `result` field as `{ traceId }`. That lets you anchor
  feedback or scores back to the exact run (e.g. `createFeedback({ traceId })`)
  without a protocol change.

Tracing is inert on Mastra cores that predate observability support, so enabling
it never breaks an older setup.
