# QA: Human in the Loop, in chat (OpenClaw)

Demo source: `src/app/demos/hitl-in-chat/page.tsx`
Route: `/demos/hitl-in-chat` · Agent: `hitl-in-chat`
Run against the real backend at `http://localhost:3119/demos/hitl-in-chat`.

Status: **supported** (tool-based HITL, the fleet's `promise-based` pattern —
see `PARITY_NOTES.md`). OpenClaw is a gateway, not a graph engine, so this uses
tool-based HITL rather than LangGraph-native `interrupt()` — that's the
intended pattern for this integration.

## What it exercises

A single HITL tool (`book_call`) defined in React with `useHumanInTheLoop`. Its
`render` draws a `TimePickerCard` inline in the chat that offers a fixed set of
candidate slots; the user's pick (or cancellation) is returned to the agent via
`respond()`, and the run continues with that choice fed back in. The tool takes
two string args — `topic` and `attendee` — supplied by the model.

OpenClaw is a single stateless gateway with no per-demo backend, so the tool is
**frontend-forwarded**: its schema rides over AG-UI in `RunAgentInput.tools`,
the ag-ui adapter hands it to OpenClaw as a caller-provided **client tool**
(the only tool list the gateway exposes to the model). When the model calls it,
the run stops on a pending tool call, the card renders in-chat, and the run only
resumes once the user submits — that resolution is the tool result.

## Prerequisites

- Stack is up; demo reachable at the URL above.
- Gateway is healthy (per-demo agent names all map to the one OpenClaw endpoint).

## Manual steps

1. Open the demo. Confirm the `CopilotChat` renders centered in a `max-w-4xl`
   container, and the two suggestion chips are visible: **Book a call with
   sales** and **Schedule a 1:1 with Alice**.
2. Click the **Book a call with sales** chip (or type: _"Please book an intro
   call with the sales team to discuss pricing."_).
3. Expect: the agent calls `book_call` and a **Book a call** card renders inline
   in the chat (`data-testid="time-picker-card"`) — a topic heading, an optional
   "With …" attendee line, and a 2-column grid of time slots.
4. Click a time slot (`data-testid="time-picker-slot"`). Expect: the card
   collapses to a confirmation state (`data-testid="time-picker-picked"`)
   reading **Booked for <slot label>**, and the agent resumes and acknowledges
   the chosen time.
5. Start a fresh request and trigger the card again, then click **None of these
   work**. Expect: the card switches to the cancelled state
   (`data-testid="time-picker-cancelled"`) and the agent resumes acknowledging
   that no time was picked.

## Assertion bar

- The card only appears after the model calls `book_call` (not eagerly).
- Buttons are interactive only while `status === "executing"`; once a slot is
  picked or the card is cancelled, all buttons disable (no double-submit).
- Exactly one resolution reaches the agent per card, and the run resumes
  coherently referencing the pick / cancellation.

## Caveats

- The candidate slots are **fixed in the frontend** (`DEFAULT_SLOTS`), not
  generated by the model — the model supplies only `topic`/`attendee`; the
  user's choice among the hardcoded slots is what flows back.
- HITL here is **tool-based** (client-tool pause + `respond()`), not LangGraph
  `interrupt()`. `gen-ui-interrupt` / `interrupt-headless` are intentionally
  not supported on OpenClaw (see `PARITY_NOTES.md`).
- Behaviour comes from the frontend + ag-ui client-tools path, not a per-demo
  backend graph — the same mechanism backs the other frontend-tool demos.
