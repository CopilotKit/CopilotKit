# Showcase

A parallel workspace under `showcase/` containing everything that powers
[showcase.copilotkit.dev](https://showcase.copilotkit.dev): per-framework
integration packages, matching starter templates, the public shell, an
alternate dojo-styled shell, and a deterministic LLM fixture server used in
tests.

Showcase is separate from the main CopilotKit monorepo (`packages/`,
`sdk-python/`, `examples/`) — it has its own `pnpm-workspace.yaml`, its own
CI jobs, and its own deployment pipeline.

## Layout

```
showcase/
  packages/        — 17 per-framework integration apps (Next.js + agent backend)
  starters/        — 17 Sales Dashboard starters (one per framework) + template/
  shell/           — public showcase.copilotkit.dev frontend
  shell-dojolike/  — alternate shell styled after the external Dojo
  aimock/          — deterministic LLM fixture server (Dockerfile + JSON fixtures)
  scripts/         — registry generation, deployment, integration scaffolding
  shared/          — shared tools (React/TS + Python), feature registry, constraints
  tests/e2e/       — global smoke specs that hit deployed Railway URLs
  INTEGRATION-CHECKLIST.md
  QA-COVERAGE.md
  STYLING-GUIDE.md
```

Three artifact types live here:

1. **Integration packages** (`packages/<slug>/`) — standalone Next.js apps, one per framework. Agent backend + demo frontend, Dockerized, deployed to Railway, and iframe-embedded in the shell.
2. **Starters** (`starters/<slug>/`) — Sales Dashboard clones users `npx degit` into their own project. Paired 1:1 with `packages/<slug>/`.
3. **Shell / aimock infrastructure** — the public site, the alternate shell, and the fixture server used by e2e tests.

## What each piece does

### `packages/<slug>/`

Standalone Next.js 15 app per framework — 17 today:
`ag2`, `agno`, `claude-sdk-python`, `claude-sdk-typescript`, `crewai-crews`,
`google-adk`, `langgraph-fastapi`, `langgraph-python`, `langgraph-typescript`,
`langroid`, `llamaindex`, `mastra`, `ms-agent-dotnet`, `ms-agent-python`,
`pydantic-ai`, `spring-ai`, `strands`.

Each package contains:

- `manifest.yaml` — slug, category, language, declared features, deployed flag
- `src/app/` — Next.js app with `/api/copilotkit` runtime route, `/api/health`, and one `demos/<feature-id>/page.tsx` per declared feature
- `src/agents/` (Python) or `src/lib/` (TypeScript) — the agent backend
- `Dockerfile` + `entrypoint.sh` — multi-stage build, starts both agent server and Next.js frontend
- `qa/` + `tests/` — manual QA checklists and Playwright specs per demo

Deployed image: `ghcr.io/copilotkit/showcase-<slug>:latest`. Live URL: `https://showcase-<slug>-production.up.railway.app`. Iframe-embedded in the shell's Integrations drawer.

### `starters/<slug>/`

Generated from `packages/<slug>/` by `scripts/generate-starters.ts`. Users pull them with `npx degit CopilotKit/CopilotKit/showcase/starters/<slug>`. One `starters/template/` drives shared content; per-framework output is regenerated — do not hand-edit generated starters.

### `shell/`

Next.js app serving `showcase.copilotkit.dev`. Integrations browser, per-demo iframe viewer, API reference (`/reference`), feature-matrix (`/matrix`), `/ag-ui`, `/docs`. Data source is `shell/src/data/registry.json`, regenerated by `scripts/generate-registry.ts`. Auto-deploys to Railway on merge to `main`.

### `shell-dojolike/`

Alternate frontend styled after the external Dojo. Part of an intentional multi-frontend consolidation rollout — lives alongside `shell/` so the two can be compared and merged without blocking production traffic. Runs on port 3001 by default.

### `aimock/`

Deterministic LLM fixture server, used by the aimock-backed e2e tests. Two fixture files:

- `feature-parity.json` — covers per-package demos (weather, pie chart, plan/steps, etc.)
- `smoke.json` — minimal fixtures for starter smoke tests

Shipped as a Docker image; also installable via `npm install -g @copilotkit/aimock`.

### `scripts/`

- `generate-registry.ts` — writes `shell/src/data/registry.json` from `packages/*/manifest.yaml`
- `generate-starters.ts` — regenerates `starters/<slug>/` from `packages/<slug>/` + `starters/template/`
- `create-integration/` — scaffolds a new `packages/<slug>/` (run via `pnpm create-integration`)
- `deploy-to-railway.ts` — image-based Railway deploy
- `bundle-demo-content.ts`, `bundle-starter-content.ts` — snapshot source files into the shell for the Code tab
- `validate-constraints.ts` — validates `shared/constraints.yaml` against `manifest.yaml`
- `capture-previews.ts` — Playwright-driven preview screenshot capture

### `shared/`

- `constraints.yaml` — per-feature requirements that manifests must satisfy
- `feature-registry.json` — the authoritative list of features across all packages
- `manifest.schema.json` — JSON schema for package `manifest.yaml`
- `typescript/tools/` — canonical implementations of the demo tools (weather, sales-todos, query-data, search-flights, schedule-meeting, generate-a2ui). Referenced by generated starters as `@copilotkit/showcase-shared-tools`.
- `python/` — Python equivalents of the shared tools plus FastAPI middleware

### `tests/e2e/`

Global smoke specs (run against deployed Railway URLs, not local builds):

- `integration-smoke.spec.ts` — `@health` / `@agent` / `@chat` / `@tools` levels per integration in the registry
- `starter-smoke.spec.ts` — same for the Sales Dashboard starter shell (STARTER_URL / STARTER env)

## Local development

Run a single package locally:

```bash
cd showcase/packages/<slug>
pnpm install
pnpm dev
```

`pnpm dev` starts the Next.js frontend and (for Python-backed packages) the agent server via `concurrently`. The frontend expects an OpenAI / Anthropic key — see the package's `.env.example`.

Running against aimock instead of a real LLM:

```bash
# One shell: aimock on :4000 with the feature-parity fixtures
npx @copilotkit/aimock --fixtures showcase/aimock/feature-parity.json

# Another shell: the package, pointed at aimock
OPENAI_BASE_URL=http://localhost:4000/v1 OPENAI_API_KEY=dummy pnpm dev
```

Adding a new framework package: see [`INTEGRATION-CHECKLIST.md`](./INTEGRATION-CHECKLIST.md). Styling rules: [`STYLING-GUIDE.md`](./STYLING-GUIDE.md).

## Testing

There are five e2e harnesses in the repo. They overlap intentionally — each catches a different failure mode.

| Harness                         | Workflow                                                                                          | What it runs                                                          | Trigger                                              |
| ------------------------------- | ------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------- | ---------------------------------------------------- |
| Legacy examples e2e             | [`.github/workflows/e2e_examples.yml`](../.github/workflows/e2e_examples.yml)                     | Playwright against `examples/**`                                      | push / PR touching `examples/**`                     |
| Per-package aimock e2e          | [`.github/workflows/showcase_aimock-e2e.yml`](../.github/workflows/showcase_aimock-e2e.yml)       | Builds one package, boots aimock, runs the package's Playwright tests | `/test-aimock <slug>` PR comment, or manual dispatch |
| Docker starter smoke            | [`.github/workflows/starter-smoke.yml`](../.github/workflows/starter-smoke.yml)                   | `docker compose up` on each starter, hits it with Playwright          | every 6 h cron + post-release + manual               |
| Build-pipeline e2e (ag-ui dojo) | [`.github/workflows/e2e_dojo.yml`](../.github/workflows/e2e_dojo.yml)                             | NX-driven Playwright against `packages/**` + `sdk-python/**`          | push / PR touching those paths                       |
| Deployed smoke monitor          | [`.github/workflows/showcase_smoke-monitor.yml`](../.github/workflows/showcase_smoke-monitor.yml) | `showcase/tests/e2e/*.spec.ts` against live Railway URLs              | cron                                                 |

### `/test-aimock <slug>` PR comment

Opt-in aimock e2e for a single package. Comment `/test-aimock langgraph-python` on a PR — the workflow resolves the slug, boots aimock with `showcase/aimock/feature-parity.json`, builds that package, and runs its Playwright suite against the aimock-backed backend. Use `/test-aimock all` for the full matrix. See the workflow above for the parsing logic.

Coverage details per demo: [`QA-COVERAGE.md`](./QA-COVERAGE.md).

## Deployment

All integration packages and both shells deploy through [`showcase_deploy.yml`](../.github/workflows/showcase_deploy.yml):

- Trigger: push to `main` touching `showcase/**`, or manual dispatch with a service selector
- Per-service job: build Docker image → push to `ghcr.io/copilotkit/showcase-<slug>:latest` → trigger Railway deploy via `RAILWAY_TOKEN`
- Live URL patterns:
  - Integration package: `https://showcase-<slug>-production.up.railway.app`
  - Starter (deployed as `starter-<slug>`): `https://showcase-starter-<slug>-production.up.railway.app`
  - Shell: `https://showcase.copilotkit.dev`

Shell auto-deploys on merge. Integration packages gate on a `deployed: true` flag in `manifest.yaml` — until that flip, the integration is listed but iframes will not render.
